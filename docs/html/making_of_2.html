An unreliable reconstruction

The Making Off - 2

Most pages on this site behave like engineers: line numbers, memory maps, source listings, proof. This page is the opposite. It is the story Sneekie might tell after 38 years in a shoebox, after enough life happened that memory became a lossy compression format.

This Is Not History. This Is Better.

I do not really remember how I wrote Sneekie. That sounds like a problem, but it is actually perfect. Accurate memories are for tax offices. Games from 1988 deserve fog, lamp light, a plastic keyboard, and a teenage certainty that POKE was not a command but a spell.

The facts are small and stubborn: Sneekie was written in July 1988, signed by HerbySoft, and printed in MS(X)DOS Computer Magazine no. 25. The rest is where the legend starts. Maybe there was a plan. Maybe there was only a snake, a maze, and the dangerous realization that video memory could be bullied directly.

Thirty-eight years later I can look at the code and recognize the handwriting of the idea, but not the evening it arrived. There were decades in between. Work. Computers. The internet. Things worth celebrating. Things that may have come in glasses. Let us not interrogate the witness too hard.

The snake was not designed. It escaped from a line number and found food.

In The Beginning There Was A Prompt

Not an AI prompt. The old kind. A blinking cursor, green and patient, asking what you wanted from the universe. In 1988 the machine did not help you. It waited. You typed. If you typed nonsense, it accepted the nonsense with professional silence and punished you later.

Sneekie probably began with the oldest programmer thought in the world: what if the screen itself is the world? No sprites. No engine. No framework. Just character cells and attributes, two bytes at a time, the way nature intended if nature had shipped with GW-BASIC.

That is how the game became a small haunted room. Hearts appeared. Stones waited to be pushed. Arrows wandered like they had somewhere important to be. The snake grew longer and more inconvenient, which is also a reasonable summary of most software projects.

The Paper Dragon

The 1988 game did not come back from a disk. It came back from paper. A printed magazine listing is both a miracle and a trap: the code is there, but it is trapped inside ink, aging paper, scan blur, and the optimistic belief that OCR can read a BASIC listing like a normal sentence.

One does not simply OCR a BASIC listing. A human sees 0 and O and thinks, yes, probably different. The scanner sees two small ovals and chooses chaos. Was that ZORE or Z0RE? Was the loop variable an O, a zero, or a tiny portal to debugging?

Every wrong character mattered. A magazine article can survive a typo. A BASIC program cannot. Change one digit in a line number and the snake walks into a wall in another dimension. Misread a colon, and suddenly the program has opinions. The OCR did its best, but the 0/O feud was the final boss.

So the recovery became detective work. Compare the scan. Read the line. Run the port. Watch it fail. Look again. Whisper to the page. Consider coffee. Consider stronger archaeology. Eventually the old listing became text again, and the text became code that could move.

Maybe This Was The Real Trick

Sneekie survived because it was simple enough to print, but strange enough to remember. It was not a file in a cloud, not a repository, not a release artifact with a changelog. It was a snake in a magazine, waiting for somebody patient enough to type it back into the future.

The beautiful absurdity is that the original readers had already done the first restoration in 1988. They bought the magazine, opened the listing, and typed page after page by hand. Every home became a compiler farm with one operator, one keyboard, and no undo stack. The meme had not been invented yet, but the spirit was there: this is fine.

Then 2026 Got Dramatic

The modern version should have been straightforward: take an old GW-BASIC program and make it run on the web. Of course it was not straightforward. The original game did not have a modern game model. It had video memory. The screen was the state. The walls, hearts, arrows, stones, score line, and snake body all lived where the old PC expected them to live.

Then, for only a few days, Claude Fable appeared like a comet in the tooling sky. In this legend, Fable did the impossible first translation: the original BASIC became a browser game, not by politely redesigning it, but by keeping the old soul intact. The port still thinks in bytes, cells, line numbers, and POKEs. It did not become a new game wearing an old hat. It became Sneekie again.

After that, Claude Opus built the house around it: the manual, source pages, magazine scans, visualizers, explanations, live bot, and the kind of website a 1988 type-in game absolutely did not ask for and absolutely deserved.

So now the game exists in three times at once: 1988, where it was written; the printed magazine, where it slept; and 2026, where a browser, some AI, and a suspicious amount of nostalgia convinced it to wake up.

The Moral, If A Snake Game Needs One

Maybe the true story is lost. Maybe it was never very cinematic. Maybe it was just a young programmer, a keyboard, and enough curiosity to turn text-mode memory into a maze. That is enough.

The rest can be legend. After 38 years, Sneekie still moves. It still eats hearts. It still punishes impatience. And somewhere between the magazine ink and the browser canvas, the little snake looked at all our modern machinery and quietly said: all your base address are belong to me.
